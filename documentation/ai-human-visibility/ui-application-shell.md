# Factory Tour: `ui.application-shell`

The application shell is the front door, the control room, and the traffic coordinator of the product. If you imagine the app as a factory, the shell is not one conveyor belt; it is the central floor where every conveyor becomes visible to a human operator. In a healthy system, this module should feel boring to the user and deeply informative to the engineer. The user should simply experience a responsive app that appears ready quickly, while the developer should see clear transitions: what hydrated first, what was deferred, what was blocked, and what eventually became visible.

Historically, shell modules in AI-assisted projects tend to absorb too much logic because they are easy to edit and highly visible. The result is usually a giant orchestration surface that can still "work," but becomes difficult to reason about when timing issues appear. This factory tour exists to prevent that blindness. The shell should be documented and instrumented as a sequence manager: it does not own every domain behavior, but it is responsible for assembling them in user-safe order and exposing predictable interaction points.

A typical run begins when the app first executes and the shell enters bootstrap mode. During this phase, the shell is deciding what must be known immediately and what can be loaded asynchronously. It needs enough state to render a trustworthy initial posture: control availability, mode gating, and basic session list context. It should avoid waiting for expensive detail work that does not affect immediate user trust. The shell's first responsibility is therefore ordering, not completeness. In telemetry terms, the first events should read like staged milestones, not random function traces.

From the user's perspective, the shell should transition through three intuitive states: loading intent, interactive readiness, and background hydration. In loading intent, the app signals that boot has started and does not pretend full readiness yet. In interactive readiness, essential controls become usable and the primary value path is unblocked. In background hydration, secondary enrichments such as detailed previews and diagnostics context arrive progressively. If these boundaries blur, users perceive "laggy mystery behavior" rather than planned sequencing.

The shell is also the module where mode boundaries are enforced. Developer mode, diagnostics surfaces, and advanced visibility toggles should be introduced here because this is where global interaction posture is managed. The shell should not implement deep diagnostics internals; it should gate and route access cleanly. Think of it as the receptionist and badge desk at the factory entrance: it verifies which doors are available but does not operate the machinery in each room.

Another crucial shell concern is synchronization banners and transitional messaging. The product may be doing legitimate work after first paint, but if users cannot see what class of work is in progress, they interpret the delay as failure. The shell's observability strategy must therefore include user-facing status narratives that remain human and concise. This is not verbose debug text; this is trust-preserving posture. A good shell tells users what kind of work remains without leaking internal complexity.

The shell is also where cross-domain failure handling can become coherent or chaotic. If capture initialization fails, if settings hydration stalls, or if session list reads return slowly, the shell decides whether the user sees a controlled fallback state or a fragmented partial UI. For this reason, shell telemetry should include not only success milestones but failure branch decisions. The debugging question is often not "did an error happen?" but "which fallback path did we choose and why?"

From a factory-tour perspective, these are the primary stations inside `ui.application-shell`: bootstrap sequencing, readiness gate, mode gating, deferred hydration orchestration, global notices, and fallback routing. A healthy trace should let an engineer walk those stations in order and understand where time was spent. An unhealthy trace usually shows interleaved or duplicated transitions, missing "done" milestones, or visible UI changes happening before required prerequisites.

Expected high-value shell signals should include bootstrap start, first interactive state reached, essential settings loaded, session list baseline loaded, deferred hydration start, deferred hydration completion, and any fallback activation. Frequency should remain low per activation, usually single-digit to low double-digit events, because shell traces should summarize phases rather than emit high-frequency noise. If shell logging becomes chatty, it usually means domain-level details are leaking upward and should be moved down to their owning modules.

When interpreting shell logs, sequence matters more than absolute count. A healthy startup usually has monotonic phase progression with minimal reversals. A warning pattern is phase oscillation, such as repeated "start" events without completion or repeated attempts to render the same readiness state. Another warning pattern is user controls becoming active before prerequisite state is confirmed, followed by immediate rollback. That usually indicates race conditions in bootstrap ordering.

For manual debugging, the human operator should run a simple interaction script: cold open the app, wait for first interaction readiness, trigger one primary action (such as opening a session or starting capture), then observe whether the shell reports clean phase continuity. If it does not, the next inspection target is usually the boundary between shell and whichever backend module failed to provide expected timing or state guarantees.

The shell's persisted object footprint is intentionally small compared to backend modules, but it still exposes crucial derived state: mode flags, sync banner posture, selected-view context, and currently active flow segment. Those should be inspectable as summary state snapshots in diagnostics tools, because many "the app feels weird" complaints come from shell state mismatch rather than broken backend persistence.

In summary, the `ui.application-shell` tour is about confidence choreography. It coordinates startup truth, interaction gating, and mode access so users can trust what they are seeing. Its observability should tell a short, ordered story of readiness and fallback decisions. If the shell can do that consistently, the entire factory becomes easier to operate, because every downstream module starts from a stable and visible entrance condition.
