# Factory Tour: backend.capture-domain

This is the factory tour of the backend capture domain. In practical terms, this module is where ephemeral microphone signal gets transformed into durable session artifacts that the rest of the system can trust. If the capture domain is healthy, everything downstream becomes dramatically simpler. If it is weak, every later stage has to compensate for uncertain timing, uncertain persistence, or uncertain integrity. For that reason, this module is both a value creator and a foundational risk surface.

In product terms, the user expectation is simple: press Start, speak, press Stop, and trust that what was spoken is safely represented in storage. In architecture terms, that expectation hides a lot of moving parts. Signal arrives in chunks that are not inherently aligned with user semantics. Device callbacks are not guaranteed to be smooth. Resource setup has asynchronous latency. Storage operations can be delayed. If capture does not continually reconcile these realities into stable artifacts, the user interface may still look active while value is quietly being lost.

From a factory perspective, the capture domain has three major zones: intake, transformation, and handoff. Intake is where the module acquires signal and confirms readiness. Transformation is where raw signal is segmented and packaged into durable units. Handoff is where those units are persisted with enough metadata for downstream playback, analysis, retention, and transcription operations. Instrumentation should mirror these zones so that humans can see where progress is happening and where it is not.

The history and context for this module is that capture reliability is often falsely assumed once a recording button exists. In real systems, the subtle failures are rarely “start completely failed.” More common are partial failures: startup lag that shifts expected timing, chunk generation that appears active but is starved, persistence backlog that eventually catches up too slowly, or graceful stop paths that leave dangling session states. The visibility contract for this module exists specifically to make these hidden failures visible before they become user trust failures.

The mechanism story begins with capture request intent. A healthy sequence starts when capture is requested and the module transitions from idle to starting with explicit state. Then stream acquisition succeeds and configuration details are captured. Once the signal path is active, the module begins accumulating enough samples to produce a durable unit. Every produced chunk is both a data artifact and a heartbeat. Chunk cadence tells you whether the module is alive. Chunk metadata quality tells you whether the module is coherent. Stop sequence completes when all pending work is flushed, persisted, and reflected into finalized session status.

Why this module is tricky is that it combines time-sensitive behavior with asynchronous dependencies. Timing can drift if capture duration uses wall-clock assumptions instead of accumulated sample counts. Persistence can race with stop transitions if flushes are not awaited deterministically. Error behavior can fragment if stream failures and encode/persist failures are treated as one generic class. Good observability requires event granularity that respects these distinctions. Without that granularity, all failures look like “recording weirdness,” which is not actionable.

Signals to watch should be organized by lifecycle phase. In intake, track request, stream acquired, and capture graph ready events. In transformation, track chunk encoded with sequence and size posture. In handoff, track chunk persisted and downstream profile/store outcomes. In finalize, track stop requested, flush completion, and session reconciliation outcome. These events should include bounded details: session id, sequence index, timestamps, durations, byte sizes, and state transitions. They should avoid huge payload dumps because high-frequency capture events can otherwise overwhelm log usability.

A healthy sequence example looks like this: capture request appears, stream acquisition appears shortly after, first chunk encode and persist appear within expected startup window, then a repeating cadence of encode/persist events continues with monotonic sequence indexes and plausible byte sizes. On stop request, a final flush appears, followed by reconciliation and idle transition. If any of these checkpoints are missing, the module may still appear superficially active while silently dropping value.

Failure cues and likely causes can be categorized. If request appears without stream acquisition, likely causes are permissions, device constraints, or initialization regressions. If stream acquisition appears but no chunk cadence follows, likely causes are callback starvation, threshold configuration mismatches, or blocked transformation loops. If encode appears but persist does not, likely causes are storage queue backpressure or handoff errors. If stop appears without reconciliation, likely causes are unawaited flush operations or state transition races. A good operator should be able to infer these classes quickly from timeline shape, not by reading source code.

For debugging operations, this module should support focused toggling so the operator can isolate capture without full-system noise. In practical incidents, humans often need to inspect capture together with storage handoff and one UI flow module, but not with unrelated transcription chatter. Module-level toggles and filterable event families are therefore not optional convenience; they are operational requirements for fast diagnosis.

The object-inspection side for this module is equally important. Capture health is validated not only by events but by artifact shape: chunk sequence continuity, monotonic time windows, and session summary consistency. A human should be able to open chunk/session records and quickly verify whether metadata reflects the event timeline. If events claim successful persistence while artifacts show missing or contradictory sequence windows, the issue is likely in handoff integrity.

The long-term value of this visibility design is that it transforms capture from a black box into an inspectable conveyor line. New contributors can reason about where a recording failed without reading internals first. AI assistants can consume structured timelines and propose targeted remediation rather than broad guesses. Most importantly, users get more reliable outcomes because the team can detect and fix silent capture degradation before it becomes a recurring production complaint.
