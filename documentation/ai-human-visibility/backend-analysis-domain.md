# Factory Tour: backend.analysisDomain

If the capture domain is the plant floor where raw material enters, the analysis domain is the quality-lab wing where the plant decides what the material means. It is where a long session stops being “just audio” and becomes a structured timeline: quiet windows, boundaries, candidate snips, and confidence that timing is coherent enough for downstream behaviors. This domain is deceptively easy to underspecify because people often treat analysis as a pure math utility. In reality, in this app, it is a stateful and iterative operating loop that mixes data completeness checks, timing verification, selective regeneration, and cache discipline.

The historical context for this domain matters because analysis was not introduced as a first feature; it emerged as sessions became longer and downstream workflows needed meaningful segmentation. The earliest form of this capability was a direct pass over frames, but once sessions reached larger sizes and retention behavior began to remove raw audio in places, “analyze now” turned into a multistep orchestration problem. You cannot trust segmentation if timing is inconsistent; you cannot trust timing if chunk duration evidence is missing; you cannot trust cached results if source profiles changed since last pass. The analysis domain therefore became a coordinator of data trust, not only an algorithm.

The mechanism story begins with a request such as “prepare analysis for this session.” At this point, the domain checks initialization posture, then resolves timing confidence through verification. If verification reports missing prerequisites, the domain may trigger regeneration of absent volume profile records from whatever chunk artifacts are still available. Then it retries verification, because now prerequisites may have changed. Only after timing reaches an acceptable state does the domain build an ordered timeline of frames. Even that step is not trivial: profiles can represent partial or cumulative contexts, so frame windows must be interpreted carefully to avoid duplicated early-session influence. Once a coherent frame timeline is built, segmentation logic runs and produces boundaries and snip candidates. Finally, the domain caches results keyed on session and data freshness signals to avoid repeating expensive work unnecessarily.

The reason this module is tricky is that it sits exactly between “storage truth” and “user-visible behavior.” If it over-trusts stale data, users see snip boundaries that feel wrong. If it under-trusts and recomputes everything constantly, performance suffers. If it skips timing verification, playback and transcription can inherit subtle drift. If it treats missing profiles as fatal, user experience degrades for recoverable sessions. This is why the analysis domain should be narrated as a control system with checkpoints, not as a single function call.

Another subtle challenge is that analysis quality depends on cross-domain assumptions that are easy to violate during refactors. Capture might change chunk cadence. Storage might alter retention semantics. Playback might reinterpret ranges differently. When any of those move, analysis assumptions can quietly diverge. The visibility plan in this domain therefore has to expose assumptions explicitly: what timing basis was used, how many chunk durations were verified, whether regeneration happened, what cache key drove reuse, and whether segmentation was produced from full or partial frame context.

Signals to watch in this module should follow the control loop shape. You need entry/exit spans for “prepare analysis,” explicit outcomes for verification, clear counts for missing prerequisites, and branch events for regeneration paths. You also need timeline assembly indicators, such as frame totals and duration limits applied, plus segmentation output stats (number of boundaries, candidate segments, expected duration vs observed coverage). Cache behavior should be visible too: cache hit, cache miss, and invalidation reasons. Without this, you cannot tell whether repeated latency is from algorithm cost or cache bypass.

A healthy sequence usually looks like this: analysis request starts; verification reports verified or quickly resolvable gaps; any needed regeneration completes with small target counts; verification succeeds; frame timeline builds with expected coverage; segmentation outputs boundaries/snips within plausible range; result is cached and returned. A second request on unchanged data should often show a cache hit, minimal recomputation, and fast completion. If those conditions hold consistently, this domain is likely operating correctly.

A fragile sequence has recognizable signatures. One common failure cue is repeated verification churn with recurring missing IDs that never converge; this often indicates regeneration cannot access required chunk data or retention has removed audio needed for profile reconstruction. Another cue is timeline inflation, where observed frame duration appears much larger than session expectations; this can point to cumulative-profile misinterpretation or offset handling errors. A third cue is cache thrash, where almost every request invalidates cache due to noisy freshness keys; this can mean update stamps are being touched too broadly by unrelated writes.

There is also a human-facing failure mode that appears outside logs first: users report that snip boundaries “feel random” or transcript retry targets look misaligned with what they heard. When that happens, this module’s tour should begin with verification outcomes and timeline coverage metrics before any tuning attempts. If the underlying temporal substrate is wrong, threshold tuning will only hide symptoms.

From an AI-to-human collaboration perspective, this module benefits from narrative diagnostics as much as from raw counters. Exporting “missingChunkIds count: 7” is useful; exporting “verification could not converge because all 7 missing profiles correspond to purged chunk audio” is transformative. The first is telemetry; the second is explanation. The standard in this repository should expect both.

The object inspection strategy for this module centers on derived artifacts and their provenance rather than only final snip outputs. Humans should be able to inspect: verification results, chunk/profile coverage maps, timeline stats, and segmentation summaries. Raw frame arrays can be huge, so summary-first rendering is important, but raw access remains necessary for deep incidents. The right experience is layered: compact health card first, then expandable detail.

If this module is healthy, it makes the rest of the system feel smart and stable. If it is opaque, the rest of the system looks flaky even when capture and storage are doing exactly what they should. That is why this domain deserves a long tour. It is the interpretation core, and interpretation cores require explicit observability if users are expected to trust outcomes.
